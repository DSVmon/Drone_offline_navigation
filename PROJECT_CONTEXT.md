# Drone Offline Navigation: Phase 4.2 (Procedural Caves & Physical Turns)

Этот файл содержит полное описание текущего состояния проекта автономного спасательного дрона для обеспечения преемственности в новых сессиях разработки Trae.

## 🎯 Цель проекта
Разработка автономной системы навигации дрона в закрытых пространствах (пещерах) с использованием стереозрения (OpenCV) в качестве основного сенсора и лазерного dToF в качестве резервного.

## 🛠 Технический стек
- **ОС:** ROS 2 Humble
- **Симулятор:** Gazebo Classic
- **Языки:** C++ (навигация/перцепция), Python (управление)
- **Библиотеки:** OpenCV 4.x, `cv_bridge`, `image_geometry`, `message_filters`
- **Среда:** Прямой запуск в Ubuntu 22.04 (WSL2) с поддержкой аппаратного ускорения (vGPU). **Docker удален** для повышения производительности графики.

## 🏗 Архитектура системы (Ноды)
1. **`navigation_node` (C++):** Обрабатывает стереопару, вычисляет карту диспаратности и определяет дистанцию в 5 зонах (Left, Center, Right, Top, Bottom). Публикует данные в `/navigation_node/stereo_distances`.
2. **`perception_node` (C++):** Обрабатывает данные с лазерных дальномеров dToF для предотвращения "ослепления" стереокамеры вблизи стен.
3. **`control_node` (Python):** Логика поведения. Использует конечный автомат миссии. **Реализует гибридное 3D-управление**: X, Y и Yaw через `planar_move`, а Z через сервис Gazebo.

## 🚀 Ключевые технические решения (Phase 4.2)

### 1. Процедурная генерация мира
- **`scripts/procedural_cave.py`:** Генерирует случайную извилистую пещеру при каждом запуске. 
- **Препятствия:** Добавлены сталактиты и сталагмиты (блоки `Gazebo/Wood`) для проверки 3D-маневрирования.
- Пещера полностью закрыта (стены, потолок, тупик в конце).

### 2. Гибридная 3D физика (Z-axis Workaround)
- **Проблема:** Стандартные плагины ROS 2 Humble (типа `planar_move`) не поддерживают ось Z, а `force_based_move` отсутствует в базовой поставке.
- **Решение:** Использование `libgazebo_ros_planar_move.so` для горизонтального движения и прямое управление координатой Z через сервис `/gazebo/set_entity_state` в `control_node.py`.
- **Результат:** Дрон полноценно маневрирует в 3D, облетая препятствия сверху и снизу. Код оптимизирован для минимизации задержек при вызове сервиса.

### 3. Миссия "Исследование и возврат"
- **SEARCHING / RETURNING:** Унифицированные состояния полета. Используют идентичную логику `_calculate_smooth_3d_speed` и порогов обнаружения препятствий (0.8м).
- **INSPECTING:** Остановка перед сложным препятствием (< 0.8м), выбор пути и анти-застревание. Использует `previous_mission_state` для корректного возврата к прерванной задаче (поиск или возврат домой).
- **TURNING:** Разворот на 180 градусов. Использует `entry_yaw` (курс входа в тупик) для обеспечения точного выхода в сторону начала пещеры с допуском 0.1 рад.
- **COMPLETED:** Остановка при достижении радиуса 0.4м от точки старта (0,0).

### 4. Стабильность и плавность (Phase 4.2.1)
- **Фильтрация сенсоров (EMA):** Внедрено экспоненциальное скользящее среднее (alpha=0.5) для данных стереозрения. Это устраняет рывки управления при кратковременных сбоях в расчете диспаратности.
- **Оптимизация скорости:** Вес центрального сенсора в расчете `stable_dist` увеличен до 80%, что снижает ложные торможения у боковых стен. Минимальная скорость поднята до 0.25 м/с для уверенного движения.
- **Гистерезис состояний:** Порог выхода из `INSPECTING` установлен на 1.5м (при входе 0.8м), что исключает циклическое "дерганье" на границе обнаружения препятствия.

### 5. Оптимизация и Логирование
- **Flight Logging:** Автоматическая запись параметров полета в CSV (`/logs/flight_*.csv`) на частоте 5 Гц. Включает 3D координаты, состояния и сенсоры.
- **Collision Detection:** Обработка данных с бампера для экстренного маневра отскока.

## ⚠️ Решенные проблемы (Lessons Learned)
- **Stereo Blindness:** При подлете ближе 0.3м стереокамера выдает ложные 10м. Исправлено через `dtof_alert` override в `control_node`.
- **Frozen Z:** Был заблокирован на уровне 1.0м из-за плагина `planar_move`. Исправлено гибридным методом (X/Y/Yaw через плагин, Z через сервис).
- **Missing Plugins:** Плагин `force_based_move` отсутствует в стандартном ROS 2 Humble. Использование `planar_move` является наиболее стабильным решением.
- **Performance:** Переход с Docker на WSL2 позволил избавиться от пропусков кадров (Message Filter dropping) и лагов симуляции.
- **Sensor Jumps:** Обнаружены резкие скачки дистанции стереокамеры в узких местах. Решено через EMA-фильтрацию в `control_node`.
- **Return Jerking:** Дрожание при полете назад было вызвано потерей контекста миссии после облета препятствия. Решено через `previous_mission_state` и полную унификацию кода SEARCHING/RETURNING.
- **Turning Precision:** Недостаточная точность разворота (0.2 рад) вызывала рывки в начале пути домой. Допуск ужесточен до 0.1 рад.
